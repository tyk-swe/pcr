# RUSTSEC-2024-0436 remediation

## Status and ownership

- Audit date: 2026-07-23
- Status: blocked on a vetted upstream release
- Internal remediation target: 2026-09-15
- Existing exception expiry: 2026-10-12 UTC
- Responsible area: Native Networking Senior Owner
- Primary owner from repository metadata: `@tyk-swe`

`RUSTSEC-2024-0436` is an informational advisory for the unmaintained
`paste` procedural macro. There is no patched `paste` release. The repository
exception remains narrowly scoped to this advisory while the preferred
upstream remediation is completed.

## Verified dependency path

The root lockfile and the all-feature Cargo graph contain this complete reverse
dependency path:

```text
paste v1.0.15 (proc-macro)
└── netlink-packet-core v0.8.1
    ├── netlink-packet-route v0.30.0
    │   └── rtnetlink v0.21.0
    │       └── packetcraftr v0.4.0-beta.1
    ├── netlink-proto v0.12.0
    │   └── rtnetlink v0.21.0
    └── rtnetlink v0.21.0
```

This was verified with:

```console
cargo tree --locked --all-features -i paste
cargo tree --locked --all-features -i netlink-packet-core
```

The path is Linux-only and enabled by the `native-route` feature. `paste` runs
while compiling `netlink-packet-core`; it is not linked into the PacketcraftR
binary.

## Existing exception and enforcement

`deny.toml` ignores only `RUSTSEC-2024-0436` and records the affected locked
version and the 2026-10-12 expiry. The dependency-policy CI job:

1. fails after the internal 2026-09-15 target;
2. retains the original hard exception expiry on 2026-10-12; and
3. runs the pinned `cargo-deny 0.20.2` policy check.

Neither date may be extended to accommodate an unresolved dependency update.
The ignore must be removed only after `cargo tree` proves that `paste` is
absent.

## Candidates evaluated

| Candidate | Advisory removal | Compatibility and impact | Decision |
| --- | --- | --- | --- |
| `rtnetlink 0.21.0` with `netlink-packet-core 0.8.1`, `netlink-packet-route 0.30.0`, `netlink-proto 0.12.0`, and `netlink-sys 0.8.8` | No | These are the latest compatible releases and already satisfy the project's features. `rtnetlink`, route, and proto declare Rust 1.66.1, 1.77, and 1.75 respectively; core and sys do not declare an MSRV, so the complete graph was checked with Rust 1.96. All three paths through `netlink-packet-core 0.8.1` retain `paste`. There is no published `rtnetlink 0.22.0`. | Retain temporarily. |
| Upgrade `netlink-packet-route` to 0.31.0 | No | Its Rust 1.77 requirement is compatible with the MSRV, but `rtnetlink 0.21.0` requires `netlink-packet-route ^0.30`. Cargo rejects 0.31.0, and 0.31.0 still requires `netlink-packet-core ^0.8.0`, which resolves to paste-bearing 0.8.1. | Rejected: incompatible and ineffective. |
| Pin `netlink-packet-core` to 0.8.0 and resolve `pastey 0.1.1` | Yes | The lockfile changes only the two package records. PacketcraftR source, features, and public API do not change. Linux platform tests, including loopback route lookup and name/index identity checks, passed, as did no-default, native-route, all-feature, and Rust 1.96 checks. However, this is a downgrade to a release that upstream deliberately superseded in 0.8.1 because it considered the `pastey 0.1` replacement insufficiently proven. It also cannot select the current `pastey 0.2` line. | Rejected: test-compatible but not a sufficiently vetted supply-chain remediation. |
| Upgrade `netlink-proto` or `netlink-sys` independently | No | 0.12.0 and 0.8.8 are already the latest releases. `netlink-proto` still requires `netlink-packet-core ^0.8.0`; `netlink-sys` does not remove the other direct core paths from `rtnetlink`. | Rejected: no effective upgrade exists. |
| Use current upstream Git revisions, a local patch, or vendoring | No verified benefit | `netlink-packet-core` upstream `main` is still the 0.8.1 release and retains `paste`. The repository denies unknown Git sources and has no accepted local patching or vendoring policy. A source override would increase review and lockfile risk without an upstream release contract. | Rejected. |
| Replace `rtnetlink` or implement route netlink locally | Potentially | This would change safety-sensitive Linux route parsing, namespace-aware connection handling, multipath selection, source selection, and interface identity behavior. It would require Rust changes and substantially more validation than a dependency-only update. | Contingency only. |

The 0.8.0 experiment used an isolated copy of the worktree. It did not alter
the committed lockfile.

Weekly Dependabot checks now cover both Cargo packages. Dependabot opens review
pull requests only: no auto-merge is configured, and major updates are not
included in the minor-and-patch groups.

## Preferred remediation

The preferred remediation is a published `netlink-packet-core` release that
removes `paste` through an upstream-reviewed implementation, followed by the
smallest compatible `rtnetlink` family update. An upstream change may use a
current, reviewed token-pasting implementation or remove token pasting from the
generated accessors entirely.

Milestones:

1. By 2026-08-01, open or confirm an upstream issue and offer a tested change
   that removes `paste` without changing the netlink packet API.
2. By 2026-08-15, recheck published versions of `rtnetlink`,
   `netlink-packet-core`, `netlink-packet-route`, `netlink-proto`, and
   `netlink-sys`; test any compatible release in isolation.
3. By 2026-09-01, if no acceptable release is available, begin the contingency
   review for a maintained published netlink stack. Do not add a Git override,
   vendor a fork, or weaken `cargo-deny`.
4. No later than 2026-09-15, land the reviewed remediation, verify that
   `cargo tree --locked --all-features -i paste` finds no package, and remove
   both the advisory ignore and its date guards.

The final change must preserve:

- IPv4 and IPv6 route lookup, local/on-link/gateway classification, multipath
  handling, preferred-source validation, and route MTU selection;
- interface name/index identity and stale-identity rejection;
- the `tokio_socket`-only `rtnetlink` feature selection;
- no-default, `native-route`, and all-feature builds;
- the public PacketcraftR API; and
- Rust 1.96 compatibility.

Required final validation:

```console
cargo tree --locked --all-features -i paste
cargo check --locked --no-default-features
cargo check --locked --no-default-features --features native-route
cargo check --locked --all-targets --all-features
cargo test --locked --all-features 'net::platform::'
cargo +1.96.0 check --locked --all-targets --no-default-features
cargo +1.96.0 check --locked --all-targets --all-features
cargo deny check
git diff --check
```
