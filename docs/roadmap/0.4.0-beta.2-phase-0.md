# PacketcraftR 0.4.0-beta.2 Phase 0

- Status: active
- Last reviewed: 2026-07-23
- Senior-review assignments:
  [review ownership](../governance/review-ownership.md)

## Release objective

The objective of `0.4.0-beta.2` is native-path stabilization, not feature
expansion. Phase 0 establishes the scope, contract freeze, ownership
boundaries, and evidence expected before parallel implementation work begins.

Work should reduce risk in existing live execution paths: deterministic
failure behavior, bounded work, resource cleanup, platform parity, and
regression coverage. A change is not in scope merely because it is adjacent to
a live command.

## Stabilization scope

The following existing live commands are in scope:

| Command | Stabilization focus |
| --- | --- |
| `send` | Route materialization, link-mode dispatch, native transmission, policy enforcement, and error propagation. |
| `capture` | Backend startup, queue limits, frame accounting, interruption, shutdown, and cleanup failures. |
| `exchange` | Capture-before-send ordering, deadlines, response correlation, unmatched evidence, and exactly-once cleanup. |
| `replay` | Captured-frame fidelity, link-mode selection, timing and traffic budgets, partial I/O, and fail-atomic accounting. |
| `scan` | Bounded probe execution, response evidence, classification, timeout behavior, and native adapter failures. |
| `traceroute` | Probe identity, hop deadlines, response classification, completion evidence, and native adapter failures. |
| `dns` | Existing UDP live execution, correlation, retry and timeout evidence, and native adapter failures. |

Cross-cutting work may cover the existing Linux, macOS, and Windows backends;
the `live`, `native-route`, `native-layer2`, and `native-layer3` feature
profiles; interface identity; route selection; libpcap/Npcap integration; raw
Layer 3 I/O; privilege failures; cancellation; and cleanup. Changes must remain
targeted to an observed stabilization risk.

## Explicitly out of scope

Phase 0 and the `0.4.0-beta.2` stabilization cycle do not include:

- new protocols;
- new CLI commands;
- output v2;
- an async runtime migration;
- broad platform rewrites;
- crates.io publication.

Feature proposals should be recorded for a later milestone instead of being
combined with stabilization fixes.

## Temporary beta contract freeze

The following contract surfaces are frozen until `0.4.0-beta.2` is released:

- `schemas/packetcraftr.packet.v1.schema.json`;
- `schemas/packetcraftr.output.v1.schema.json`;
- the `packetcraftr.output/v1` aggregate and stream envelope implemented under
  `src/output/envelope/` and its command/mode vocabulary under
  `src/output/contract/`;
- the `packetcraftr.protocol-support/v1` manifest structure implemented in
  `src/protocol/support/manifest.rs`;
- all published packet and output documents under `examples/documents/`.

Existing fields may not be removed, renamed, made newly required, or
semantically changed during stabilization. Existing schema identifiers,
envelope discriminators, enum spellings, command names, event names, required
field sets, and manifest section meanings are also frozen. A change to Rust
serialization that changes emitted data is a contract change even when the
JSON Schema would still accept it.

Only backward-compatible optional additions may be considered. They require
approval from the Contract Senior Owner identified in the
[review-ownership policy](../governance/review-ownership.md). Approval must be
recorded on the pull request and must cover:

1. the compatibility rationale and why the stabilization fix cannot avoid the
   addition;
2. schema, output-model, and published-example consistency;
3. aggregate and streaming behavior where both exist;
4. validation of every published document and a regression test for the new
   optional field.

Adding an enum or discriminator value, changing a default, reinterpreting an
existing value, or changing field presence is not treated as an optional field
addition. It remains out of scope for this freeze.

## Parallel-development boundaries

Each pull request has one primary responsibility. Mechanical refactoring and
behavior changes travel in separate pull requests so reviewers can establish
whether behavior changed. A pull request that crosses two senior-review
boundaries needs approval from each affected owner before merge.

The sensitive boundaries and escalation rules are defined in
[review ownership](../governance/review-ownership.md). Contributors should
claim an issue before making overlapping edits and use the repository area,
type, and priority labels described in [CONTRIBUTING.md](../../CONTRIBUTING.md).

## Required evidence

A stabilization pull request must include:

- a focused regression test for the observed behavior;
- the relevant portable, default, and complete feature-profile results;
- platform-specific evidence when a native backend changes;
- deterministic failure-path tests for native networking changes, including
  cleanup or partial-failure behavior when relevant;
- confirmation that public APIs and frozen contracts did not change, or the
  required owner approval for an allowed optional addition;
- confirmation that no new source file exceeds 20 KiB or roughly 600 lines.

## Exit criteria

Phase 0 is complete when:

- the scope and ownership rules are being used for stabilization pull
  requests;
- release-blocking `priority/p0` issues for the seven in-scope commands are
  resolved or explicitly waived by the release owner;
- remaining `priority/p1` issues are resolved or assigned a documented
  follow-up;
- changed native paths have success and relevant failure-path regression
  coverage;
- the portable, default, and all-feature CI profiles remain green on their
  supported platforms;
- the frozen schemas, output envelope, protocol-support manifest structure, and
  published documents remain compatible;
- the existing GitHub binary-release workflow is validated without adding
  crates.io publication.

## Baseline already present

The repository already provides versioned packet and output schemas, typed
output-envelope models, a centralized built-in protocol catalog, a versioned
protocol-support manifest, published contract documents, schema/example
regression tests, multi-platform CI, fuzz smoke tests, and a tag-driven GitHub
binary release workflow. `Cargo.toml` also has `publish = false`. Phase 0
governance builds on those mechanisms rather than replacing them.
