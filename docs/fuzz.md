# Bounded field-aware fuzz workflow

`fuzz` deterministically mutates the reflective fields of one packet recipe.
Offline build and dissection is the default and has no resolver, route,
capture, neighbor, or transmission seam. Live execution is available only
with `--live`; every built case then uses the same traffic policy, route
planner, exact builder, capture-readiness barrier, response matcher, bounded
evidence, and joined cleanup lifecycle as `exchange`.

Only transmit fuzz cases in an isolated environment where you have explicit
authorization. Public destinations remain denied unless
`--allow-public-destinations` is present. A permissive or deliberately
malformed case needs both `--allow-malformed-live` at the operation boundary
and `--allow-permissive-packets` in traffic policy. These checks cover the
complete campaign and occur before an interface is resolved or a route,
capture provider, neighbor resolver, or sender observes a packet.

```console
# Reproducible offline mutation of every reflective field.
packetcraftr --output json fuzz \
  --packet 'ipv4(src="192.0.2.1",dst="192.0.2.2")/udp(sport=40000,dport=9)/raw(text="hello")' \
  --seed 42 --cases 64

# Restrict mutation to the payload and reproduce absolute case 19.
packetcraftr --output ndjson fuzz \
  --packet 'ipv4(src="192.0.2.1",dst="192.0.2.2")/udp(sport=40000,dport=9)/raw(text="hello")' \
  --seed 42 --first-case 19 --cases 1 \
  --strategy bit-flip --field 2.bytes

# Deliberately exercise dependent lengths in a private lab.
packetcraftr fuzz \
  --packet 'ipv4(src="192.168.56.1",dst="192.168.56.2",identification=1)/udp(sport=40000,dport=9)/raw(text="hello")' \
  --strategy malformed --field 1.length --mode permissive --live \
  --allow-malformed-live --allow-permissive-packets \
  --interface lab0 --timeout-ms 250 --rate 10
```

## Determinism and strategies

The operation seed and absolute case index derive an independent 64-bit case
seed. Case `N` does not depend on generating cases `0..N-1`, so the
`operation_seed`, `case_index`, and `case_seed` reproduction tuple can select a
single failure directly with `--first-case N --cases 1`. The same packet,
ordered strategies, field targets, seed, index range, build mode, and limits
produce the same ordered mutations and offline bytes.

`--field LAYER.FIELD` uses the zero-based layer index from the packet recipe.
It may be repeated or comma-separated; without it, fields are enumerated in
layer and schema order. Strategies may likewise be repeated or comma-separated:

- `boundary` selects deterministic type-aware empty, zero, one, maximum,
  address, control-text, byte, and list boundaries. A protocol-specific range
  may reject a wider reflective candidate; that rejection is retained as a
  case result rather than panicking.
- `random` generates a seeded value for the field's reflective `FieldKind`.
  Text, byte, and list allocation never exceeds the configured per-field
  bounds.
- `bit-flip` changes one deterministic bit of byte fields. Oversized existing
  byte fields are reduced to the configured field bound before mutation.
- `malformed` targets fields marked derived by the layer schema and replaces
  automatic lengths, checksums, offsets, or discriminators with exact or raw
  values. Strict mode normally rejects inconsistent cases; permissive offline
  mode retains their diagnostics and marks them as requiring live opt-in.

Each case includes the attempted mutation, original and replacement tagged
`FieldValue`, complete representable packet document (the unchanged recipe
when `set_field` rejects the attempted value), deterministic shrink values,
exact built bytes when successful, bounded decode result, diagnostics,
reproduction tuple, and classified rejection when mutation or build fails.
Shrink values are data, not an implicit search: callers can apply them to the
reported `layer` and `field` and run the resulting recipe through their own
predicate.

## Bounds and live evidence

`--cases`, `--max-cases`, snap length, aggregate generated bytes,
per-field bytes, list elements, shrink steps, capture queue frames/bytes,
timeout, rate, and maximum duration are all finite. Checked arithmetic rejects
case-index, byte, duration, evidence, and statistic overflow. The base packet
must also fit the builder layer limit and the hard 4,096-reflected-field
selection ceiling before any case clone. Mutation and
offline building finish before live authorization. Retained original/mutated
values, recipes, shrink data, decoded fields, and exact built frames share the
aggregate generated-byte ceiling even when a case is rejected. The
conservative live wire
budget includes a possible synthesized Ethernet envelope and must fit both the
fuzz byte limit and the shared traffic-policy packet/byte limits before the
first route lookup.

Live cases execute one at a time through `Client::exchange`. Exact sent bytes,
matcher-correlated responses, decoded unmatched traffic, undecodable frames,
capture loss counters, diagnostics, and timeouts remain distinct. Response,
unmatched, and undecodable retention share one campaign-wide evidence budget;
omission produces `fuzz.evidence_limit` without changing a response outcome
into a timeout. Invalid executor frames, byte counts, capture counters, or
successful-send statistics fail the operation as internal evidence errors.

Text, aggregate JSON, and NDJSON render the same typed result. Text passes
mutation values, protocol errors, diagnostics, and captured data through the
common terminal-control and bidirectional-control escaping boundary. JSON
preserves tagged values and exact hexadecimal bytes. NDJSON emits one `case`
record per generated case followed by one `complete` record with final counts,
diagnostics, and operation statistics.

## Rust API

`FuzzRequest`, `FuzzLimits`, `FuzzTarget`, and `FuzzStrategy` describe portable
generation. `fuzz` exposes only the offline packet and registry inputs, making
network side effects impossible through that API. `fuzz_live` additionally
requires injectable `FuzzAuthorizer`, `FuzzExecutor`, and `FuzzClock` seams.
The root façade supplies `TrafficPolicyFuzzAuthorizer` and
`ClientFuzzExecutor`; `SystemFuzzClock` supplies production rate timing.
External codecs participate through their normal `LayerSchema`, `field`,
`set_field`, builder, dissector, and matcher implementations without a
fuzz-specific registry or unchecked allocation path.
